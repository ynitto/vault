# kiro-loop 並列実行と操作性改善

## 概要
`kiro-loop` の並列実行制御、起動安定性、対話操作をまとめて改善したセッション群から、実装方針と失敗パターンを抽出した。

## ポジティブな学び
分散動作を崩さずに、ファイルベースの排他とプレーンテキスト前提の UI 制御へ寄せると安定しやすい。

- **分散セマフォはファイルロックで十分機能する**: `~/.kiro/slots/` と `fcntl.flock` を使う `GlobalSemaphore`、実行終了を検知してスロットを返す `SlotMonitor` により、統合管理プロセスを追加せずに同時実行数を制御できた。
- **tmux/kiro-cli は TUI より classic モード前提が扱いやすい**: `--trust-all-tools` による確認ダイアログを避け、`--classic` と許可プロンプト自動応答に寄せると、`kiro-loop` 側の待機・送信ロジックを簡潔に保てた。
- **設定探索順の明示は運用負荷を下げる**: `<cwd>/.kiro/` → `<cwd>/` → `~/.kiro/` → `~/` の順で見る設計、`cwd` 指定、`DEFAULT_CONFIG_NAMES` 許容により、ワークスペース運用とホーム共通設定を両立できた。
- **CLI 操作は小さな標準機能で十分改善できる**: `readline` の導入だけで左右移動・上下履歴を実現でき、`kiro-loop ls/send` の追加で `kiro-send` の別実装を吸収できた。
- **macOS 対応は OS 判定とツール差分を先に潰すと進めやすい**: `Darwin` 許可、BSD/GNU `sed -i` 差異吸収、`brew install` 案内追加のように、失敗点を列挙してから直す方が手戻りが少なかった。

## ネガティブな学び（失敗・回り道）
途中停止したリファクタや、プロンプト形式を決め打ちした待機ロジックは壊れやすい。

- **中断された大きなリファクタは参照切れを生みやすい**: `KiroSession` 統合途中で未定義参照や旧クラス参照が残り、まず壊れた状態の棚卸しからやり直す必要があった。
- **`--trust-all-tools` 前提は TUI に引きずられる**: テキスト入力前提の自動化に対して確認ダイアログが割り込み、起動失敗やタイムアウトの原因になった。
- **プロンプト検出を単純な行末一致で書くと詰まる**: `3% >` や `2% > Want to know what commands I have?` のような実際の出力に追従できず、正規表現修正と許可プロンプト処理が必要になった。

## 結論・決定事項
`kiro-loop` は分散セマフォ + classic モード + 明示的な設定探索順を基本方針にするのがよい。大きな整理は「壊れた状態の復旧」→「小さな機能追加」→「再リファクタ」の順で進める方が安全。
