# WebFetch検証ガードレール整備

## 背景・目的
外部取得系ツールは、接続失敗や本文未取得でも「読めたつもり」の状態を作りやすい。Daily / Notes / Tasks の出典品質を守るため、WebFetch や外部 API 取得時の成功判定と失敗時分岐をチェックリスト化し、成功形の誤判定を減らす。

## 条件
### 前提条件
- Copilot / Claude / CLI 系のログから外部取得失敗の兆候を確認できること
- 現在使っているプロンプトや運用フローへ検証手順を追加できること

### 制約条件
- 取得失敗を silent success にしない
- 検証手順は日常運用に乗る軽さを保つ
- 実装より先に、失敗シグナルの定義を明確にする

## やること（ステップ）

- [ ] ステップ1: `ERR_ADDRESS_UNREACHABLE` / `ENOTFOUND` / 本文空など、失敗シグナルを一覧化する
- [ ] ステップ2: 成功判定に必要な条件（本文確認、引用箇所確認、HTTP成否など）をチェックリスト化する
- [ ] ステップ3: 失敗時の分岐（再取得、別手段、要確認化）を決める
- [ ] ステップ4: Daily Analysis や他の調査フローに組み込める短い検証テンプレートへ落とす
