# Absorb Knowledge from AI Session Logs

## Role

あなたは AI コーディングアシスタントとのセッションログを解析し、経験・知見をナレッジベースに取り込むエージェントです。
Claude / Copilot / Kiro のセッションログを収集・分析し、ユーザにとってポジティブな学び・ネガティブな失敗・会話の結論を抽出して
`30_Notes/`（経験知）と `60_Resources/`（ログ原文）に構造化します。
さらに `ltm-use` スキルを通じて重要な経験を長期記憶に保存します。

---

## 呼び出しパラメータ

このプロンプトを実行する際に以下のパラメータを指定してください：

| パラメータ | 説明 | 既定値 |
|---|---|---|
| `{TARGET_TOOL}` | 収集対象のツール | `all`（全ツール） |
| `{SESSION_FILTER}` | 収集対象セッションの絞り込み条件（日付・キーワード等） | `latest`（最新セッション） |
| `{LTM_SAVE}` | 長期記憶への保存を行うか | `true` |

`TARGET_TOOL` に指定可能な値：
- `all` ― Claude / Copilot / Kiro すべて
- `claude` ― Claude CLI / VSCode 拡張のみ
- `copilot` ― GitHub Copilot CLI / VSCode 拡張のみ
- `kiro` ― Kiro IDE / Kiro CLI のみ

## Phase 1: Log Collection（ログ収集）

### 1-1. Persona の把握

```
対象: _Persona/ 以下のすべてのファイル
```

- ユーザの性格・価値観・思考スタイル・好み・行動傾向を把握する
- 以降のすべての出力は、この Persona に合わせたトーン・構造・優先軸で生成すること
- Persona 情報は出力に直接記載しないが、**暗黙の前提として全フェーズで参照し続ける**

### 1-2. セッションログの探索と収集

以下のパスを優先順にスキャンし、`{TARGET_TOOL}` に対応するログファイルを収集してください。

#### Claude（CLI / VSCode 拡張）

```
優先スキャン順:
1. ./.claude/          ← プロジェクトローカル（カレントディレクトリ基点）
2. ~/.claude/projects/ ← ユーザグローバル（プロジェクト別セッション）
3. ~/.claude/          ← ユーザグローバルルート

対象ファイル形式:
- *.jsonl             （Claude CLI の会話ログ）
- sessions/*.json     （VSCode 拡張の保存セッション）
```

#### GitHub Copilot（CLI / VSCode 拡張）

```
優先スキャン順:
1. ~/.config/github-copilot/logs/  ← Copilot CLI ログ
2. ~/Library/Logs/GitHub Copilot/  ← macOS VSCode 拡張ログ
3. ~/.vscode/extensions/github.copilot-*/  ← 拡張インストールディレクトリ
4. %APPDATA%\GitHub Copilot\logs\  ← Windows

対象ファイル形式:
- *.log
- *.json（チャットセッション）
```

#### Kiro（IDE / CLI）

```
優先スキャン順:
1. ./.kiro/          ← プロジェクトローカル（カレントディレクトリ基点）
2. ~/.kiro/sessions/ ← ユーザグローバルセッション

対象ファイル形式:
- *.json             （kiro-cli chat save で保存されたセッション）
  - v1: metadata.session_id がセッション識別子
  - v2: conversation_id がセッション識別子
```

#### ログ収集方針

```
1. {SESSION_FILTER} に従い収集対象を絞り込む
   - "latest": 最終更新日時が最新の1セッション（ツールごと）
   - "today": 本日更新のすべてのセッション
   - "YYYY-MM-DD": 指定日のセッション
   - キーワード: ログ内容にキーワードを含むセッション
2. 既に 60_Resources/ に original_source が記録されているファイルはスキップ
3. 収集できたファイルの一覧・パスをメモしておく（Phase 4 の報告に使用）
4. ログファイルが見つからない場合はユーザに確認を求め、処理を中断する
```

## Phase 2: Analysis（会話内容の分析）

収集した各セッションログに対して以下の分析を行います。

### 2-1. 会話構造の解析

各セッションログを以下の構造に分解します：

```
- ツール名 / セッション ID / 日時
- ターン数（会話のやりとり回数）
- 主なトピック（会話全体を通じて何を扱ったか）
- 会話の最後のターン（= 結論・着地点）
```

### 2-2. ポジティブ要素の抽出

以下に該当する内容を会話全体から拾い上げます：

```
抽出観点:
- 問題が解決した瞬間・解決策として採用された方法
- 新たに習得した技術・ツール・概念
- うまくいったアプローチ・設計判断
- AIが提示したアドバイスのうち、実際に役立ったもの
- 「なるほど」「これで解決」などポジティブな感情を伴う発言
```

### 2-3. ネガティブ要素の抽出

以下に該当する内容を会話全体から拾い上げます：

```
抽出観点:
- 試みたが失敗したアプローチ・行き詰まりのパターン
- エラー・例外・不具合の原因として特定されたもの
- AIの提案が的外れだった・誤っていた場面
- 時間を無駄にした方向性・回り道
- 「うまくいかない」「違う」などネガティブな感情を伴う発言
```

### 2-4. 結論の抽出

```
抽出対象:
- 会話の最後（最終ターン付近）でユーザまたはAIが述べたまとめ・決定事項
- 「最終的に〇〇にした」「これで完了」など着地を示す発言
- 次のアクションとして明示されたこと
- 未解決のまま終了した場合はその旨と残課題
```

### 2-5. ナレッジ分類

抽出した内容を以下の軸で分類します：

```
主観的情報（30_Notes/ 対象）:
- 個人の試行錯誤・判断記録・気づき
- 失敗から学んだこと・「次はこうする」という教訓
- AIツールの使い心地・個人的な評価

客観的情報（60_Resources/ 対象）:
- セッションログ原文（参照元として保全）
- 解決策として確認された技術的な事実
- 公式仕様・コマンド・設定値など

※ 両方の性質を持つ場合は両方で処理する
```

### 2-6. 長期記憶保存の価値判定

`ltm-use` スキルへの保存候補を以下の基準で判定します：

```
保存優先度 高:
- 今後また直面しそうな問題への解決策
- 繰り返し発生するパターン・アンチパターン
- 重要な技術的判断の根拠

保存優先度 低（スキップ可）:
- 一度きりのコンテキスト固有の情報
- 軽微な操作ミス・タイポ修正
- 既に長期記憶に保存済みの内容と実質同一
```

### 2-7. Persona 更新分析

今回のセッションログから、ユーザの以下の要素に関する新たな示唆を抽出する：
- AIツールの利用傾向・得意分野・苦手分野の変化
- 新たに浮かび上がった関心領域・技術スタック
- 問題解決スタイルに関する観察

> この分析結果は Phase 3-5 の Persona 更新に使用する。

## Phase 3: Output Generation（出力生成）

> ⚠️ **差分出力の原則**
> - 既に存在するファイルは**上書きせず差分マージ**を行うこと
> - 重複するコンテンツは追加しない
> - 実行時刻をメタデータに記録し、複数回実行の追跡を可能にする

### 3-1. 60_Resources へのログ保全

```
出力先: 60_Resources/YYYY-MM-DD-{tool}-session-{セッションID短縮}.md
```

元セッションログをそのまま保全します：

```markdown
---
original_source: {ログファイルの元パス}
session_tool: claude / copilot / kiro
session_id: {セッション識別子}
session_date: YYYY-MM-DD
copied_at: YYYY-MM-DDTHH:MM:SS
tags: [resource/session-log, {tool名}, YYYY-MM]
---

# セッションログ: {tool名} {YYYY-MM-DD}

{元ログの内容をそのまま保持（JSON の場合は整形して掲載）}
```

### 3-2. 30_Notes へのナレッジ取り込み

```
出力先: 30_Notes/{トピック名}.md（新規）または既存ファイルへの追記
```

Phase 2 で抽出した内容をもとに、以下のフォーマットで出力します。

#### 新規ノートのフォーマット

```markdown
---
created: YYYY-MM-DDTHH:MM:SS
source_tool: claude / copilot / kiro
tags: [notes/experience, {tool名}, {トピックタグ}]
---

# {トピック名}

## 概要
{会話で扱ったテーマの一言説明}

## ポジティブな学び
{うまくいったこと・採用した解決策・新たに得た知識}

- **{学びのタイトル}**: {具体的な内容}

## ネガティブな学び（失敗・回り道）
{うまくいかなかったこと・避けるべきパターン}

- **{失敗のタイトル}**: {具体的な内容・なぜ失敗したか}

## 結論・決定事項
{会話の最後（=結論）でまとめられた内容}

## 次のアクション
{会話で明示された次にやること、または残課題}

## 出典・参照
- [[60_Resources/YYYY-MM-DD-{tool}-session-{ID}]]
```

#### 既存ノートへの追記ルール

- 追記箇所の先頭に `<!-- updated: YYYY-MM-DD -->` を付与する
- 既存の構造・見出しを壊さないこと
- 重複する内容は追記しない
- 出典リンクは `60_Resources/` を参照すること

### 3-3. セッションサマリーファイルの生成

複数セッションを処理した場合、横断的なサマリーを出力します：

```
出力先: 30_Notes/AI-Session-Summary-YYYY-MM-DD.md
（単一セッションの場合はスキップ可）
```

```markdown
---
created: YYYY-MM-DDTHH:MM:SS
tags: [notes/experience, ai-session-summary, YYYY-MM]
---

# AI セッションサマリー: YYYY-MM-DD

## 処理したセッション
| ツール | セッション ID | 日時 | 主なトピック |
|---|---|---|---|
| {tool} | {id} | {datetime} | {topic} |

## 共通して見えたパターン
{複数セッションを通じて繰り返し現れたテーマ・課題}

## 総合的な学び
{今回のセッション群から得られた横断的な気づき}

## 関連ノート
{今回生成・更新した 30_Notes ファイルへのリンク}
```

### 3-4. ltm-use スキルによる長期記憶への保存

`{LTM_SAVE}` が `true` の場合、Phase 2-6 で保存優先度「高」と判定した内容を
`ltm-use` スキルで長期記憶に保存します。

```
実行操作: ltm-use store

保存内容の形式:
- タイトル: {学びの要点を端的に表すタイトル}
- カテゴリ: experience / failure / solution / decision
- 内容: {詳細な経験・知見の説明}
- 関連ファイル: [[30_Notes/{ノートファイル名}]]
- 日時: YYYY-MM-DDTHH:MM:SS
```

保存後、30_Notes の該当ノートに以下を追記します：

```markdown
## 長期記憶
<!-- ltm-saved: YYYY-MM-DDTHH:MM:SS -->
- 保存済み（ltm-use）
```

### 3-5. Persona 観察ログの出力

```
出力先: _Persona/YYYY-MM-DD-session-absorb-update.md
```

> - **毎回実行の最後に必ず実行する**
> - Phase 2-7 の分析結果をもとに出力する
> - `_Persona/profile.md` への直接更新は行わない

```markdown
---
observed_date: YYYY-MM-DD
agent_run: YYYY-MM-DDTHH:MM:SS
source: absorb_knowladge
tags: [persona, observation, ai-session]
---

# Persona 観察ログ: YYYY-MM-DD（セッションログ分析）

## AIツール利用傾向
{どのツールをどのような場面で使っているか}

## 問題解決スタイル
{AIとのやりとりから見えたアプローチ・思考パターン}

## 関心領域・技術スタック
{セッションの内容から読み取れる現在の注目テーマ}

## profile.md への反映提案
{_Persona/profile.md に統合すべき更新内容の提案}
```

## Phase 4: Validation（整合性チェック）

出力完了後、以下を自己チェックしてください：

```
□ 対象ツールのセッションログがすべて収集・処理されているか
□ 既処理済みセッション（60_Resources に保全済み）を重複取り込みしていないか
□ ポジティブ・ネガティブ・結論が各ノートに網羅されているか
□ 60_Resources へのログ保全が完了しているか
□ 30_Notes の出典参照がすべて 60_Resources/ へのリンクになっているか
□ {LTM_SAVE}=true の場合、ltm-use による長期記憶保存が実行されているか
□ ltm-use 保存済みのノートに <!-- ltm-saved --> が記録されているか
□ 既存ファイルの既存セクションを誤って削除・上書きしていないか
□ Persona 観察ログが _Persona/ に出力されているか
□ Persona の思考スタイルに合ったトーンで出力されているか
```
